Das WORDL Plugin-System verstehen

Diese Seite erklaert, wie ein externes JAR zur Laufzeit neue Wortlaengen in WORDL bereitstellt: vom Plugin-Vertrag ueber URLClassLoader und ServiceLoader bis zum atomaren Austausch des Wortkatalogs.

1
Ein Plugin ist ein normales Java-JAR mit einer Klasse, die ein gemeinsames Interface implementiert.
2
Das Backend scannt den Plugin-Pfad alle 10 Sekunden und baut daraus einen neuen Katalog-Snapshot.
3
Spiel-Anfragen lesen immer einen vollstaendigen Katalog: entweder den alten oder den neuen.

Grundidee

Ein Plugin liefert Daten, nicht Spielregeln

In WORDL erweitert ein Plugin nicht die Rate-Logik. Es liefert nur zusaetzliche Loesungswoerter fuer genau eine Wortlaenge. Der Typ des Plugins ist deshalb keine Zeichenkette wie "seven-word", sondern direkt die Anzahl der Buchstaben.

Plugin-Typ

type() == 7

Das Plugin stellt siebenbuchstabige Woerter bereit. Diese Zahl taucht spaeter als neue Wortlaenge in der API und im Frontend auf.

Plugin-Daten

words()

Die Methode liefert eine Liste von Woertern. Das Backend normalisiert auf Grossbuchstaben und ignoriert ungueltige Eintraege.

Spiel-Logik

bleibt im Backend

Bewertung von Buchstaben, Versuche, Status und API bleiben Teil der Hauptanwendung. Das Plugin bekommt dort keine Kontrolle.

Das ist eine bewusste Architekturentscheidung: Der Plugin-Vertrag ist klein. Dadurch koennen Plugins leicht gebaut werden und die Hauptanwendung bleibt verantwortlich fuer die Regeln.

Architektur

Welche Bausteine zusammenspielen

Das Plugin-System besteht aus wenigen Klassen. Jede Klasse hat eine klare Aufgabe: Vertrag definieren, JARs finden, Plugins laden, Woerter zusammenfuehren und die verfuegbaren Wortlaengen ausliefern.

Plugin-JAR

Liegt neben dem Spiel-JAR oder in wordl.plugin.dir.

WordPluginLoader

Findet JAR-Dateien und laedt Provider mit ServiceLoader.

WordListPlugin

Gemeinsames Interface zwischen Hauptanwendung und Plugin.

WordCatalog

Fuehrt eingebaute Woerter und Plugin-Woerter zu einem Snapshot zusammen.

GameController

Liefert /api/games/word-lengths und startet Spiele.

Vertrag im Backend 
public interface WordListPlugin {
    int type();

    List<String> words();
}
Gedankliches Datenmodell

Nach dem Laden sieht der Katalog intern etwa so aus:

4 -> [BARK, CALM, ...]
5 -> [APPLE, BRICK, ...]
6 -> [ANCHOR, BEACON, ...]
7 -> [BALANCE, CAPTURE, ...]  // aus Plugin

Theorie

Warum ein ClassLoader ueberhaupt noetig ist

Normalerweise kennt Java nur Klassen, die beim Start auf dem Classpath liegen. Ein Plugin-JAR wird aber spaeter in einen Ordner kopiert. Die laufende Anwendung muss deshalb zur Laufzeit neue Klassen aus diesem JAR laden koennen. Genau dafuer wird ein URLClassLoader verwendet.

Plugin-ClassLoader

Laedt aus Plugin-JARs

SevenLetterWordsPlugin.class

META-INF/services/...

Parent-ClassLoader

Laedt die Hauptanwendung

WordListPlugin.class

WordCatalog.class

Parent Delegation

Gemeinsame Typen kommen aus der App

Wenn das Plugin WordListPlugin braucht, wird diese Klasse vom Parent-ClassLoader der Hauptanwendung verwendet. So meint das Plugin dasselbe Interface wie das Backend.

Provided Dependency

Backend nicht ins Plugin packen

Das Plugin braucht das Backend-JAR nur zum Kompilieren. Zur Laufzeit stellt die Hauptanwendung das Interface bereit. Deshalb ist <scope>provided</scope> richtig.

Wuerde das Plugin eine eigene Kopie von WordListPlugin mitbringen, koennte Java zwei gleich benannte, aber unterschiedliche Typen sehen. Dann waere eine Plugin-Klasse unter Umstaenden nicht als WordListPlugin der Hauptanwendung verwendbar.

ServiceLoader

Wie Java herausfindet, welche Klasse ein Plugin ist

Das Backend koennte alle Klassen in einem JAR durchsuchen. Das waere aufwendig und ungenau. Stattdessen nutzt Java einen Standardmechanismus: ServiceLoader. Das Plugin sagt in einer kleinen Textdatei, welche Klasse den Service implementiert. Diese Datei liegt im Plugin-JAR unter META-INF/services und muss exakt so heissen wie das Service-Interface.

JAR-Struktur
seven-word-0.0.1-SNAPSHOT.jar
at/easyshop/wordl/plugins/seven/SevenLetterWordsPlugin.class
META-INF/services/de.demo.wordl.backend.plugin.WordListPlugin
Service-Datei

Der Dateiname ist der vollqualifizierte Name des Interfaces: de.demo.wordl.backend.plugin.WordListPlugin. Die Datei enthaelt den vollqualifizierten Klassennamen der Implementierung: 

at.easyshop.wordl.plugins.seven.SevenLetterWordsPlugin
1. Datei lesen

META-INF/services

Der ServiceLoader sucht im Plugin-JAR nach einer Datei, die exakt wie das Interface heisst: META-INF/services/de.demo.wordl.backend.plugin.WordListPlugin.

2. Klasse laden

URLClassLoader

Der Klassenname aus der Datei wird mit dem Plugin-ClassLoader geladen.

3. Instanz erzeugen

provider.get()

Java erzeugt ein Objekt und das Backend ruft type() und words() auf.

Lifecycle

Laden, Entladen und der atomare Katalogtausch

Der Plugin-Pfad wird beim Start und danach alle 10 Sekunden gescannt. Dabei wird kein bestehender Katalog schrittweise veraendert. Stattdessen entsteht ein neuer Snapshot, der am Ende auf einmal veroeffentlicht wird.

Alter Snapshot

Request-Threads lesen weiter

4 5 6
Neuer Snapshot

Nach dem Scan sichtbar

4 5 6 7
Warum volatile?

Der Scheduler schreibt eine neue Map-Referenz, waehrend HTTP-Requests sie lesen. volatile sorgt dafuer, dass die neue Referenz fuer andere Threads sichtbar wird.

Leser sehen dadurch nicht halb gefuellte Daten. Sie sehen entweder den alten vollstaendigen Snapshot oder den neuen vollstaendigen Snapshot.

Was bedeutet entladen?

Wenn ein JAR verschwindet, liefert der naechste Scan keine Woerter mehr aus diesem Plugin. Der neue Snapshot enthaelt diese Wortlaenge dann nicht mehr.

Die JVM entlaedt Klassen nicht sofort garantiert. Da das Backend aber nur Strings speichert und keine Plugin-Instanzen behaelt, kann der alte ClassLoader spaeter vom GC aufgeraeumt werden.

Im Log werden echte Zustandswechsel sichtbar: Laden, Entladen und Reload einer Plugin-Wortliste. Reine Scans ohne Aenderung erzeugen keine Info-Meldung.

Praxis

Was ein Plugin-Projekt konkret braucht

Ein Plugin ist ein eigenes Maven-Projekt. Es kompiliert gegen das Backend-Interface, packt das Backend aber nicht ein. Das fertige JAR wird in den Plugin-Ordner gelegt.

Maven Dependency 
<dependency>
    <groupId>de.demo.wordl</groupId>
    <artifactId>backend</artifactId>
    <version>0.0.1-SNAPSHOT</version>
    <scope>provided</scope>
</dependency>
Implementierung 
public class SevenLetterWordsPlugin
        implements WordListPlugin {

    public int type() {
        return 7;
    }

    public List<String> words() {
        return List.of("BALANCE", "CAPTURE");
    }
}
Checkliste fuer Schueler
  • Das Plugin liegt in einem eigenen Projekt ausserhalb des WORDL-Backends.
  • Die Klasse implementiert WordListPlugin.
  • type() entspricht exakt der Wortlaenge.
  • Die Service-Datei liegt unter META-INF/services.
  • Das gebaute JAR liegt im Plugin-Ordner oder neben dem Spiel-JAR.
Typische Fehler
  • Falscher Klassenname in der Service-Datei.
  • Backend-JAR ins Plugin gepackt statt provided verwendet.
  • Woerter haben nicht exakt die Laenge aus type().
  • Das JAR wurde kopiert, aber die 10 Sekunden bis zum naechsten Scan wurden nicht abgewartet.
  • Das Plugin startet eigene Threads und beendet sie beim Entfernen nicht.

Hintergrund

Warum diese Architektur fuer Unterricht gut passt

Kleiner Vertrag

Wenig Magie im Plugin

Das Plugin liefert nur Daten. Dadurch bleibt der Einstieg ueberschaubar, obwohl echte Java-SPI-Technik verwendet wird.

Echte JVM-Konzepte

ClassLoader greifbar machen

Schueler sehen, warum Klassen zur Laufzeit geladen werden muessen und warum gemeinsame Interfaces wichtig sind.

Nebenlaeufigkeit

Snapshot statt Mutation

Der atomare Referenztausch zeigt eine einfache Strategie fuer sichere Reads waehrend Hintergrundjobs schreiben.

Classpath
Die Menge an Orten, aus denen Java beim Start Klassen laden kann.
ClassLoader
Ein Objekt, das Klassen findet, laedt und in der JVM verfuegbar macht.
URLClassLoader
Ein ClassLoader, der Klassen und Ressourcen aus URLs laden kann, hier aus lokalen JAR-Dateien.
ServiceLoader
Java-Standardmechanismus, der Implementierungen eines Interfaces ueber META-INF/services findet.
Snapshot
Eine vollstaendige, unveraenderliche Sicht auf den aktuellen Wortkatalog.
Entladen
Fachlich: Plugin-Daten verschwinden aus dem Katalog. Technisch: Klassen koennen spaeter mit dem ClassLoader vom GC entfernt werden.
